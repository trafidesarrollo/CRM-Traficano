# PRIORIDAD 1 — Seguridad y permisos

## Problemas a corregir

* eliminar cualquier fallback inseguro como `userId = 1`
* eliminar secrets default inseguros
* revisar todas las rutas backend
* implementar control real de autenticación y autorización

## Quiero que implementes

### 1. Middleware de autenticación

Crear middleware `requireAuth` que:

* valide sesión o token
* rechace acceso no autenticado
* adjunte usuario autenticado al request

### 2. Middleware de autorización por rol

Crear middleware `requireRole` o equivalente para roles:

* admin
* gerente_comercial
* vendedor
* operador

Aplicarlo a rutas sensibles:

* configuración IA
* Gmail connections
* importaciones
* usuarios
* auditoría
* cambios masivos
* prompts
* seguimiento automático

### 3. Política de acceso por entidad

Definir acceso por rol:

* admin: acceso total
* gerente: acceso a toda la operación comercial
* vendedor: solo sus clientes, oportunidades, actividades y mails asignados
* operador: carga y corrección, pero no configuración crítica

### 4. Secretos

* obligar a usar variables de entorno seguras
* si falta una variable crítica, la app debe fallar con error claro
* no usar `SESSION_SECRET` default inseguro
* no hardcodear credenciales ni tokens

### 5. Seguridad de tokens Gmail

* revisar almacenamiento de tokens
* guardar refresh token y access token de forma segura
* minimizar exposición en logs
* no mostrar tokens en frontend ni respuestas API

### 6. Logs de seguridad

Registrar:

* logins
* logouts
* intentos fallidos
* cambios de permisos
* conexiones/desconexiones Gmail

# PRIORIDAD 2 — Gmail real y operable

## Problemas a corregir

* hoy la sincronización está limitada y simplificada
* el envío de respuestas no es real
* no hay manejo serio de threads
* no hay manejo serio de refresh token ni expiración
* no hay soporte útil para adjuntos

## Quiero que implementes

### 1. Conexión Gmail con OAuth 2.0 real

* revisar flujo OAuth actual
* asegurar almacenamiento correcto de refresh token
* soportar reconexión de cuenta
* mostrar estado de conexión:

  * conectada
  * vencida
  * error
  * reconectar requerida

### 2. Refresh token y expiración

Implementar renovación automática de access token usando refresh token antes de sincronizar o enviar mails.

### 3. Sincronización seria

Mejorar el módulo de sync para:

* no limitarse a `is:unread`
* permitir sincronización incremental
* soportar paginación
* guardar `messageId`, `threadId`, `historyId`, fecha, labels, remitente, destinatarios, asunto, cuerpo
* detectar duplicados por `messageId`
* permitir sincronización manual y automática

### 4. Manejo de threads

* guardar hilos correctamente
* asociar mensajes a thread
* detectar si un mail pertenece a una oportunidad existente
* mantener contexto de conversación

### 5. Envío real de respuestas

El endpoint de respuesta debe:

* enviar realmente por Gmail API
* guardar el mail enviado
* asociarlo al thread
* registrar resultado real
* marcar errores de envío si ocurren

No quiero simulación ni solo cambio de estado en DB.

### 6. Adjuntos

Implementar soporte inicial para:

* detectar si un mail tiene adjuntos
* guardar metadata de adjuntos
* permitir descargar/ver adjuntos
* preparar estructura para extracción futura desde PDF/Excel/imagen

### 7. Detección de cliente/contacto

Al sincronizar un mail:

* intentar vincular remitente con cliente/contacto existente
* si no existe, dejarlo como “sin vincular”
* permitir vinculación manual posterior

# PRIORIDAD 4 — Matching técnico de medidas y productos

## Problemas a corregir

Hoy no alcanza con un `ILIKE %medida%`.
Necesito algo usable para entorno industrial B2B.

## Quiero que implementes

### 1. Pipeline de extracción separado

Separar claramente:

* clasificación de mail
* extracción de datos
* matching contra catálogo

No mezclar todo en un solo paso.

### 2. Normalización técnica

Implementar una capa de normalización para medidas:

* pulgadas
* mm
* formatos mixtos
* diámetros
* espesores
* longitudes
* normas
* abreviaturas comunes
* símbolos variantes

### 3. Matching por niveles

Cuando la IA detecta una medida o descripción, el sistema debe devolver:

* exact_match
* approximate_match
* no_match

Con score de confianza.

### 4. Resultado estructurado

Guardar por extracción:

* texto original detectado
* medida normalizada
* norma detectada
* cantidad detectada
* familia sugerida
* producto sugerido
* tipo de match
* score de confianza
* usuario que corrigió
* versión de prompt usada

### 5. Revisión manual

Permitir al usuario:

* aceptar
* corregir
* rechazar
* vincular manualmente a producto/código interno

### 6. Prompt editable bien integrado

La pantalla de prompts debe permitir:

* editar prompt
* versionar
* activar/desactivar
* probar con ejemplos
* ver JSON devuelto
* comparar con resultado esperado

### 7. Base de equivalencias

Crear soporte para equivalencias técnicas o aliases de medida/producto.
Ejemplo:

* misma medida expresada de distintas maneras
* distintas abreviaturas de norma
* unidades alternativas

# PRIORIDAD 5 — Motor real de seguimiento automático

## Problemas a corregir

Hoy no existe un follow-up engine real.

## Quiero que implementes

### 1. Modelo de datos

Crear tablas o estructuras equivalentes para:

* `followup_rules`
* `followup_events`
* `followup_templates`
* `followup_runs`

### 2. Reglas configurables

Cada regla debe poder definir:

* tipo de mail o tipo de oportunidad
* prioridad
* delay inicial
* delay siguiente
* máximo de seguimientos
* condición de cierre
* plantilla a usar
* si requiere aprobación humana o no

### 3. Scheduler / job runner

Implementar proceso que:

* detecte seguimientos pendientes
* genere evento
* envíe o prepare borrador
* pause si el cliente respondió
* cierre si corresponde

### 4. Reglas mínimas del MVP

Soportar esta secuencia:

* seguimiento a 24h
* seguimiento a 72h
* seguimiento a 7 días
* cierre a 14 días sin respuesta

### 5. Condiciones de pausa/cancelación

Si el cliente responde:

* pausar seguimiento pendiente
* actualizar estado de oportunidad
* registrar evento

### 6. Seguimiento solo para RFQ en esta etapa

No aplicar automatización profunda todavía a reclamos o preguntas.
Solo clasificarlos y separarlos.
