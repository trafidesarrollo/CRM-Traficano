Quiero que actúes como un **senior full-stack engineer / software architect** y me ayudes a **implementar la Fase 1 completa** del módulo de **Inbox Comercial / Gestión de Emails** de mi CRM B2B industrial argentino, y dejar preparado el camino para las siguientes fases.

## Contexto del producto

Estoy desarrollando un CRM full-stack para una empresa industrial B2B argentina. El problema real a resolver es que hoy usan una casilla compartida tipo `ventas@empresa.com` y el manejo de emails es desordenado: varios responden, no se sabe quién tomó cada consulta, qué quedó pendiente ni cómo hacer seguimiento comercial.

Quiero que Gmail sea solo el canal de entrada, y que el **CRM sea el lugar real donde se trabaja**.

## Objetivo de Fase 1

Quiero construir la base sólida del módulo de Inbox Comercial.

### Esta fase NO debe intentar hacer todo.

En esta fase quiero enfocarme en:

1. **Modelado de datos correcto**
2. **Sincronización de emails desde Gmail**
3. **Agrupación por conversación / thread**
4. **Persistencia robusta evitando duplicados**
5. **API mínima para listar conversaciones y ver detalle**
6. **UI mínima funcional para probar el flujo end-to-end**

---

# Concepto central

La unidad principal NO debe ser el email suelto.
Debe ser la **conversación comercial**, agrupada por `gmailThreadId`.

## Cada conversación debe representar un thread comercial y tener:

* subject
* gmailThreadId
* estado inicial
* prioridad inicial
* referencia opcional a cliente/contacto/oportunidad
* fecha del último mensaje
* cantidad de mensajes
* último inbound
* último outbound
* snippet o preview
* flags mínimas para UI

## Cada conversación tendrá múltiples mensajes

Y cada mensaje debe guardar:

* gmailMessageId
* conversationId
* direction: inbound / outbound
* from / to / cc
* subject
* bodyText
* bodyHtml
* hasAttachments
* receivedAt
* metadata útil para sync

## 3) Implementá la lógica de sync de Gmail para Fase 1

Quiero que armes una capa de servicio clara para sincronizar emails y convertirlos a conversaciones.

## Reglas funcionales de sync

Para cada mensaje traído desde Gmail:

1. obtener `threadId`
2. buscar conversación por `gmailThreadId`
3. si no existe, crearla
4. verificar si `gmailMessageId` ya existe
5. si no existe, insertar el mensaje
6. actualizar `lastMessageAt`, `lastInboundAt`, `lastOutboundAt`, `snippet`, `messageCount`
7. si es nueva conversación, crear evento inicial
8. evitar duplicados siempre

## Quiero que propongas una estructura clara tipo:

* `gmail.service.ts`
* `conversations.service.ts`
* `sync-email.service.ts`
  o una variante mejor si lo ves más limpio

## Importante

* si todavía no está lista toda la integración OAuth real, podés dejar la capa preparada y mockear la parte de fetch desde Gmail si hace falta
* pero la lógica de persistencia y agrupación debe quedar real y bien hecha

## Necesito

* funciones claras
* manejo de errores
* operaciones idempotentes
* comentarios sólo donde sumen
